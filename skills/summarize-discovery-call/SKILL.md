# /summarize-discovery-call — Post-Call Summary + Follow-Up Builder

> **Purpose.** Triggered by the `call.ended` webhook. Ingests the call recording / transcript and produces: (1) a 1-paragraph factual summary, (2) 5-7 bullet decision-relevant facts, (3) a verbatim list of objections raised, (4) the prospect's stated decision criteria, (5) a recommended next action mapped to the acquisition FSM, (6) a draft follow-up DM that references specific call moments (not generic "great chat"), (7) a draft proposal stub if a proposal is the recommended next action.
>
> **Bar.** The follow-up DM that ships from this skill must read as if the operator dictated it the moment they closed Zoom — specific, warm, referencing exact call moments, with one clear next step. The proposal stub (when applicable) must be specific enough that the operator can polish-and-send in <15 minutes rather than write-from-scratch in 90 minutes. The summary must be tight enough that the operator re-reads it 3 days later before the follow-up call and remembers the room.
>
> **Why the speed matters.** Discovery-call follow-up is a 24-hour decay function. A specific, in-voice follow-up sent within 4 hours of call end converts at 2-3× the rate of a generic follow-up sent 48 hours later. This skill's job is to compress the 60-90 minutes the operator would spend writing follow-up into 5 minutes of operator review-and-send, *without* sacrificing the specificity that makes the follow-up convert.

## When to invoke

- **Auto-triggered** by `call.ended` webhook on every `Booked → CallHeld` transition (per `workflows/divisions/acquisition.md` and `paperclip.manifest.yaml`)
- Manual invoke when a recording arrives late (operator forgot to start recording, manually uploads after the fact)
- Manual re-run when the operator updates the call disposition mid-stream ("actually, this is heading to ProposalSent — re-run with proposal-stub mode")
- Never invoke without a transcript or recording — per `spec/CONTEXT-THRESHOLDS.md`, the input is gating

## Required inputs

1. **Call recording or transcript** — primary input. Recording preferred (allows tone analysis); transcript acceptable (text-only mode). If transcript is auto-generated by tooling, speaker labels are required.
2. **Call-prep deck** — produced by `/build-call-prep` for this same prospect. Provides the trigger hypothesis to validate, the case-study anchor to confirm landed, the objections that were pre-framed.
3. **Application form data** — same intake the call-prep deck used (consistency check between what the prospect declared pre-call vs. what they articulated on the call).
4. **DM thread history** — the verbatim language thread leading up to the call (used to detect language continuity in the follow-up).
5. **Operator's calendar** — to time-box delivery (summary + follow-up draft must arrive within 60 minutes of call end so the operator can review-and-send same-day).
6. **Objection library** — for the "objection extraction → calibration" loop (extracted verbatim objections feed back to refine the library).
7. **Retainer offer doc** — from `/design-retainer-offer`, for the proposal-stub mode.

## The build sequence

### Step 1 — Transcript ingestion + speaker identification (4 min)

Ingest the recording or transcript. If recording, transcribe via the workspace's runtime adapter (Whisper / Deepgram / native). Apply speaker diarization.

```yaml
transcript_processing:
  source_type: ""                         # recording | transcript-text-only | transcript-with-labels
  duration_minutes: 0
  speaker_count_detected: 0               # should be 2 for standard discovery
  speaker_labels:
    - speaker_id: ""
      role_assigned: ""                   # operator | prospect | (other if 3+ on call — flag)
      confidence: ""                      # HIGH | MEDIUM | LOW
  transcript_quality: ""                  # CLEAN | NOISY (background) | INCOMPLETE (audio dropouts)
```

If `speaker_count_detected > 2`: flag for operator — likely a stakeholder joined (partner, co-founder, board member). This changes the `next_action_recommendation` toward stakeholder-aware paths.

If `transcript_quality = INCOMPLETE`: produce summary with explicit gap flags noting which segments were unrecoverable. Operator reviews before send.

### Step 2 — Decision-relevant fact extraction (6 min)

The decision-relevant fact set is the 5-7 things the operator needs to remember before the follow-up. Extract from transcript:

```yaml
decision_facts:
  revenue: { declared_band, confidence }                          # band verbatim if given; HIGH/MED/LOW
  team_size: { declared, relevant_roles }                         # which functions exist
  prior_agency_experience: { has_history, last_engagement_summary, pain_carryover }
  current_pain_quantification: { pain_named, quantified_cost, urgency_signal }
  decision_process: { decider, stakeholders_named, stakeholder_meeting_required, expected_decision_timeline }
  budget_signal: { range_indicated, pricing_response, confidence }
  urgency_signal: { timeline_stated, forcing_function }
  trigger_validation:
    pre_call_trigger_hypothesis: ""       # from call-prep deck
    actual_trigger_articulated: ""        # what the prospect said on the call
    match: true | false                   # if false → refines the trigger taxonomy
```

If `match: false`, the workspace logs a refinement signal for `/build-call-prep`'s trigger-event taxonomy. Cycle-N feedback loop in action.

### Step 3 — Objection extraction (4 min)

Pull every objection raised on the call. Verbatim, not summary. Verbatim is required because (a) the objection library calibrates from real prospect language, (b) the follow-up DM will reference specific objections in the prospect's own phrasing.

```yaml
objections_raised:
  - objection_quote: ""                   # verbatim from transcript — exact words, including hesitations
    objection_class: ""                   # mapped to the matrix in /build-call-prep Step 4
    when_raised: ""                       # call timestamp band — early / mid / close
    operator_response_summary: ""         # how the operator handled it
    apparent_resolution: ""               # RESOLVED | PARTIAL | UNRESOLVED — based on prospect's follow-on response
    follow_up_required: true | false
```

Per `INVARIANTS.md` A-9 — these verbatim objections are routed to the objection-library refinement queue. The library compounds from real call data, not assumed objections.

### Step 4 — Decision criteria extraction (4 min)

Distinct from objections. Decision criteria are what the prospect *positively* said they need to see/hear/believe to move forward. Often surfaced by Q4 of the discovery sequence ("If 6 months from now you said this was the best call you made…").

```yaml
decision_criteria:
  - criterion: ""                         # what they need to be true
    source_quote: ""                      # verbatim
    evidence_required: ""                 # what proof would satisfy this criterion
    addressable: true | false             # can the agency credibly satisfy this?
    addressed_on_call: true | false       # was it addressed live, or does follow-up need to address?
```

Decision criteria → drive proposal stub structure (Step 7) and follow-up DM emphasis (Step 6).

### Step 5 — Next-action recommendation (3 min)

Map to the acquisition FSM (per `workflows/divisions/acquisition.md`). The four legitimate post-CallHeld branches:

```yaml
next_action:
  recommended_state_transition: ""        # CallHeld → Closed | ProposalSent | CallObjection | NotAFit
  rationale: ""                           # 2-3 sentences supporting the recommendation
  confidence: ""                          # HIGH | MEDIUM | LOW
  branch_logic_used:
    yes_now_signals: []                   # what signals support yes-now
    proposal_signals: []                  # what signals support proposal-required
    objection_signals: []                 # what unresolved objections support objection-handling-followup
    not_a_fit_signals: []                 # what signals support honest decline
  operator_confirmation_required: true    # always TRUE per A-13 (operator runs the call, operator confirms)
```

Per `INVARIANTS.md` A-13 — the workspace recommends, the operator confirms. The recommendation is never auto-executed; it surfaces for operator one-click confirmation.

Branch decision rules:

| Branch | Trigger conditions |
|---|---|
| **CallHeld → Closed** | Prospect verbally committed on call AND budget signal confirmed AND no unresolved objections AND solo-decider OR all stakeholders on call |
| **CallHeld → ProposalSent** | Prospect signaled intent but needs written for stakeholder review OR multi-stakeholder decision process AND budget range plausible |
| **CallHeld → CallObjection** | One or more objections marked UNRESOLVED in Step 3 AND prospect did not decline outright |
| **CallHeld → NotAFit** | Mutual recognition of mis-fit OR hard-DQ surfaced live OR prospect declined outright |

### Step 6 — Follow-up DM draft (5 min)

The follow-up DM, sent within 24 hours (target: 4 hours) of call end. The DM has three jobs:

1. **Reference a specific call moment** — not "great talking" but "the part where you mentioned the in-house writer leaving — that pattern is exactly what {anchor case study descriptor} faced before we worked together"
2. **Confirm next step explicitly** — calendar link, proposal arrival window, or objection-handling resource — never ambiguous "let me know what you think"
3. **Match the prospect's language register** — formal if they were formal, casual if they were casual, direct if they were direct

DM template structure (varies by next-action branch):

```yaml
followup_dm:
  branch_template_used: ""                # yes-now-confirm | proposal-bridge | objection-handle | clean-decline
  message_body: |
    {opening that references a specific call moment — verbatim phrase from prospect or operator paraphrase of a key exchange}
    
    {middle that bridges to the next step — proposal arrival, calendar tap, or objection resource}
    
    {close that respects the prospect's stated timeline / decision process — never pushy, always specific}
  estimated_send_time_after_call_end: "<4 hours"
  voice_calibration_check: ""             # passes | fails — match against operator's voice profile
  banned_phrases_check: ""                # passes | fails — no "great call", "circle back", "touch base", "synergy"
```

Banned phrases for follow-up DMs (in addition to `spec/BANNED-VOCABULARY.md`):
- "great call" / "great chat" / "great connecting"
- "circle back" / "touch base" / "follow up" (use specific verbs)
- "as discussed" / "per our conversation" (lazy — reference the actual moment)
- "thoughts?" / "let me know" (ambiguous next step)

The DM draft is operator-edit-ready — meaning operator review takes ≤2 minutes before send, not ≤2 minutes of structural rewrite.

### Step 7 — Proposal stub draft (6 min, conditional on branch)

If `next_action.recommended_state_transition = ProposalSent`, generate a proposal stub. Pulls structure from `skills/design-retainer-offer/SKILL.md` output. The stub is NOT a full proposal — it is a populated skeleton the operator polishes.

Stub structure:

```yaml
proposal_stub:
  prospect_facing_title: ""               # named transformation from offer, customized
  page_1_problem_summary:                 # from Step 2 — verbatim pain, verbatim cost
    pain_named, cost_of_inaction, why_now: ""
  page_2_transformation_hypothesis:       # from offer doc + Step 2 prospect metric
    transformation_statement, case_study_anchor, time_horizon, success_metric: ""
  page_3_scope_and_deliverables:          # from offer doc — populated, not generic
    deliverables: []
    scope_guardrails: []                  # explicit out-of-scope
    cadence: ""                           # weekly / monthly rhythm
  page_4_investment_and_terms:
    monthly_price, contract_term_months, payment_terms, guarantee: ""
  page_5_decision_criteria_addressed:     # from Step 4 — explicit per criterion
    - { criterion, how_addressed }
  next_step: { action, timeline }         # specific date/time, not "let me know"
```

The stub references the prospect's own decision criteria (Step 4) directly — most agency proposals address generic criteria; this addresses *what this prospect actually said they needed*. That specificity is the close lever.

Per `INVARIANTS.md` N-3 — every metric/transformation/quote in the stub MUST be traceable to `case_studies.agency_case_studies` or verifiable prospect intake. No invented numbers. Per N-8 — the proposal carries a per-client transformation hypothesis (Step 2 output, customized). Generic proposals close generic clients (= bad-fit clients).

### Step 8 — Output emission

Deliver bundle to operator within 60 min of `call.ended`: call summary markdown (Steps 1-5), follow-up DM draft (Step 6), proposal stub (Step 7, if branch = ProposalSent), state-transition recommendation (one-click confirm per A-13), objection-library refinement signal (async-routed per A-9). Delivery channel: operator preference — Slack DM, email, or workspace UI.

## Validation gate (mandatory before declaring done)

Three checks:

1. **The verbatim-quote test.** The follow-up DM must contain at least one specific reference to a call moment (verbatim phrase, named exchange, or prospect-articulated insight). If the DM reads as something that could be sent to any prospect, it fails — re-draft against transcript.

2. **The next-step specificity test.** The follow-up DM must end with a specific next step (calendar link with date, proposal arrival window with timestamp, named resource with delivery commitment). "Let me know your thoughts" fails — close with action.

3. **The branch-evidence test.** The recommended state transition must cite ≥2 evidence points from Step 5's branch_logic_used. A recommendation with only 1 evidence point means the call was ambiguous and the operator should review before confirming.

## Failure modes and guards

- **Generic follow-up DM.** "Great call today — let me know your thoughts" is the default failure. Guard: validation Step 1 (verbatim-quote test) is hard-block. The DM must reference specifics or it doesn't ship.
- **Wrong branch recommendation.** Misclassifying a soft-objection as Closed-pending creates a follow-up that pushes too hard; misclassifying a Closed signal as ProposalSent loses momentum. Guard: branch decision requires ≥2 evidence points (validation Step 3). Below 2 = MEDIUM/LOW confidence flagged for operator review.
- **Inventing facts the prospect didn't say.** Most damaging in proposal-stub mode where invented metrics violate N-3. Guard: every metric / quote in the stub must trace to transcript or `case_studies.agency_case_studies`. The skill's transcript-citation system tags every claim with its source.
- **Missing call recording / poor transcript.** When transcript is incomplete, the summary degrades. Guard: explicit gap flags in output; operator reviews before send. Never produce a confident summary from incomplete data.
- **Voice contamination.** When run for a `client-call` scope (rare), follow-up DM must be in the client's voice, not the agency's. Guard: voice profile loaded based on scope; voice-calibration check (Step 6) gates send.
- **Over-eager proposal stub.** Generating a proposal stub when the call actually warranted objection-handling first burns the proposal as a "did you receive my proposal?" loop. Guard: proposal stub only generates if branch logic clearly favors ProposalSent — not if branch is CallObjection or ambiguous.
- **Ignoring stakeholder-on-call signal.** When a third speaker is detected, the follow-up must address the stakeholder's concerns separately. Guard: Step 1 flags 3+ speakers and Step 6 modifies the DM template accordingly.
- **Late delivery.** 60-min target slips to 4 hours; operator has context-switched; follow-up pushes to next day. Guard: target-window declared; runtime adapter escalates on slip.
